## 本番前にやること

- `https://.../health` が `ok` を返すことを確認する
- Cloud Run の URL を確定する
- 受付や卓に置く `共通 QR コード` を刷る
- 10 人前後で投稿リハーサルをする
- UI や導線確認は `AI_PROVIDER=mock` で通し、Gemini は少数テストと本番採点に絞る
- 会場 Wi-Fi の有無と、4G/5G の入りやすさを確認する
- 管理用 PC のテザリング or モバイルルーターを準備する
- HDMI 変換アダプタとプロジェクター接続を確認する

## Gemini free tier 運用方針

- 前提:
  - `gemini-2.5-flash` を Gemini API の Free tier で使う想定
  - 2026-03-28 時点の公式 rate limit は `10 RPM / 250,000 TPM / 250 RPD`
  - RPD は `midnight Pacific time` で reset される
  - 現設定の本番日 `2026-04-19` では、日本時間の reset は `2026-04-19 16:00 JST`
  - rate limit は project 単位で効くので、同じ project の他テスト分も消費される
  - 最終的には AI Studio 上の active limits を当日確認する
- このアプリの実装上の前提:
  - 採点は `1 投稿 = 1 API call`
  - `すでに採点済みの投稿も再採点する` を付けない限り、採点済み投稿は再度叩かない
  - 上位候補の見た目調整は再採点ではなく `点数補正` で済ませられる
- 運用ルール:
  - UI 確認、投稿リハーサル、発表導線確認は `mock` を使う
  - Gemini は「コメントの雰囲気確認の少数テスト」と「本番の全件採点」に絞る
  - `2026-04-18 16:00 JST` 以降は、Gemini の不要なテストを止めて本番日ぶんを温存する
  - `すでに採点済みの投稿も再採点する` は、prompt / model を変えたときか、全体結果が明らかに不自然なときだけ使う
  - 数件だけ失敗した場合は、`force` を付けずに再実行する
  - 途中で止まった場合も、`force` なしの再実行で `pending / failed` を拾い直して再開する
- 実験回数の目安:
  - 事前の Gemini テストは合計 `20 requests` 以内
  - 本番の全件採点は `1 回`
  - 当日の救済・再実行は `30 requests` 程度までを目安
  - 1 日の soft cap は `150 requests`
  - `200 requests` を超えそうなら、その日は free tier を攻めずに `mock` へ切り替えるか paid tier へ移行する
- 頻度制限の注意:
  - Free tier の `10 RPM` は、平均すると `約 6 秒に 1 回` が上限
  - 現在の採点処理は投稿を直列に舐める実装で、Gemini 側への明示的な rate limit 制御は入っていない
  - そのため、Gemini の応答が `6 秒 / 件` より速く続くと `429` 系の rate limit に当たる可能性がある
  - 目安として、`20 件` でも安全側に見ると `約 2 分`、`70 件` なら `約 7 分` 以上かかる見込み
  - Cloud Run の deploy メモでは request timeout を `60 秒` にしているため、投稿数が多い本番では Free tier のまま全件採点する運用は要注意
- 当日のおすすめ手順:
  1. リハーサルや見た目確認は `mock`
  2. 締切後に provider を `auto` もしくは `gemini`、model を `gemini-2.5-flash` にして全件採点を開始する
  3. 採点中は admin ページを開いたままにし、進捗表示が終わるまで閉じない
  4. 数件だけ `failed` が出たら 1 分ほど待って `force` なしで再実行
  5. 上位候補の最終調整は `点数補正` で行い、Gemini の全件再採点は避ける
  6. API が不安定なら `mock` へ戻し、必要なら上位 3 枚は人力で確定する

## Android / 端末差分への考え方

- 投稿画面は保守的なフォームにしてある
- 共通 QR から入ったあとは `新郎友人 / 新婦友人 / 新郎親族 / 新婦親族` の 4択に絞っている
- 対応ブラウザでは自動圧縮する
- 圧縮に対応しないブラウザでも通常送信にフォールバックする
- それでも送れない人が数人出る前提で、救済フローを先に決めておく

## 当日の救済フロー

1. ゲストに同じ画面でもう一度送信してもらう
2. Wi-Fi が不安定なら、モバイル回線で再挑戦してもらう
3. それでも無理なら、管理側スマホでその人の専用 URL を開いて代理投稿する
4. 最終手段として AirDrop / LINE / メールで写真を受け取り、代理投稿する

## 当日の運用順

1. 開宴前に `/admin` を開いておく
2. 投稿受付が `OPEN` であることを確認する
3. 共通 QR の掲示位置を確認する
4. 締切直前に投稿数を確認する
5. 中座時に受付を閉じる
6. `mock` での rehearsal 以外は、Gemini の全件採点は 1 回を基本にする
7. AI 採点を開始したら、進捗表示が終わるまで admin ページを開いたままにする
8. 数件の失敗は `force` なしで再実行し、上位調整は `点数補正` を使う
9. `/presentation` をフルスクリーン表示する

## トラブル時の最小プラン

- AI API が不調なら `mock` に切り替える
- 無料枠を使い切りそうなら、全件再採点はやめて `点数補正` と人力確認で逃がす
- 全件採点が間に合わなければ、投稿済みの中から上位候補を人力で 3 枚選んで発表する
- プロジェクター接続が不調なら、静止画のスクリーンショットを用意しておく
